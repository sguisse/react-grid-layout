# Architecture Platform V6.4 : Hybrid & AST-Driven Migration Engine

## 0. Overview du Système
La plateforme évolue vers un modèle de **compilateur agentique hybride**. Elle combine la puissance déterministe de scripts **Python** (pour le scan et les dépendances) avec l'intelligence de raisonnement des subagents et la précision chirurgicale du moteur **AST Babel**.

```mermaid
graph TD
    A[Repo RGL Source] --> B{Orchestrator V6.4}
    B -->|Phase 0: Python Scout| C[(Global IR)]
    C -->|Queue management| Q[.github/IR/discovery_queue.json]
    B -->|Phase 1 & 2: Analysis| D[Subagents: Test/Comp/DnD]
    D -->|Structured JSON| C
    C -->|Architecture Blueprint| E[React 19 Architect]
    E -->|AST Patches| F[AST Implementer]
    F -->|Semantic Commands| G[Babel AST Engine]
    G -->|Atomic Update| H[Target Codebase]
    H -->|Validation| I[Playwright Parity]
    I -->|Feedback Loop| B
```

---

## 1. Phase 0 : Discovery Déterministe (Python Scout)
L'orchestrateur délègue désormais la cartographie initiale à un script Python pour garantir une précision de 100% sur l'inventaire des fichiers et leurs dépendances.

### Fonctionnement :
* **Split Physique** : Séparation immédiate de `/src` et `/test`.
* **Graphe de Dépendances** : Extraction automatique des imports pour identifier les candidats `core/` (logic) vs `react/` (UI) selon les règles de `CLAUDE.md`.
* **Queue Management** : Création de `discovery_queue.json` pour suivre le statut (`pending`, `in-progress`, `done`) et permettre une reprise sur erreur.

---

## 2. Phase 1 & 2 : Contrat Immuable & Forensic Analysis
Le système sépare le "Contrat de Vérité" (le comportement voulu) de la "Dette Technique" (l'implémentation actuelle).

### Fonctionnement :
* **Immutable Contract** : Le `test-analyzer` convertit les fichiers de `test/examples/` en un schéma JSON de garanties comportementales (ex: calculs de collision, limites de grille).
* **Isolation du Cerveau** : Le `dnd-specialist` extrait les mathématiques pures (compaction "Rising Tide", calculs `calcXY`) pour les migrer vers `src/core/`.
* **Legacy Detection** : Identification automatique des patterns à supprimer (ex: `forwardRef` obsolètes en React 19).

---

## 3. Phase 3 & 4 : Synthèse V2 & Architecture `CLAUDE.md`
L'architecte projette les données de l'IR vers la cible technologique : **React 19** et **dnd-kit v0.3.2**.

### Fonctionnement :
* **Blueprint React 19** : Application des refs natives et utilisation de `DragOverlay` pour la performance 60fps.
* **Pluggable Compactors** : Mise en place des interfaces de compaction et des `PositionStrategy` définies dans `CLAUDE.md`.

```mermaid
sequenceDiagram
    participant O as Orchestrator
    participant IR as Global IR
    participant RA as React Architect
    participant CM as CLAUDE.md

    O->>IR: Read Forensic Models
    O->>CM: Read V2 Design Rules
    RA-->>RA: Map to React 19 + dnd-kit 0.3.2
    RA->>IR: Write ARCHITECTURE_BLUEPRINT & TASK_LIST
```

---

## 4. Phase 5 : AST-Driven Implementation (Babel Engine)
Le moteur Babel garantit l'intégrité du code source en manipulant des arbres sémantiques plutôt que du texte brut.

### Fonctionnement :
* **Semantic Patching** : Remplacement ciblé de nœuds (ex: transformer une classe en hook, ou injecter une nouvelle logique de collision).
* **Safety Lock** : Validation syntaxique avant écriture. Si le patch échoue, l'erreur est logguée dans l'IR pour une auto-correction immédiate.

---

## 5. Phase 6 : Validation & Parité Playwright
La migration n'est validée que si le clone se comporte exactement comme l'original sur la base des use-cases extraits.

### Fonctionnement :
* **Playwright Parity** : Le `validation-agent` génère des tests de comparaison différentielle entre l'ancien RGL et le nouveau clone.
* **Traceability Audit** : Chaque décision et chaque patch est consigné dans `executionTrace.json` pour audit ou rollback.

---

## 6. Manifeste de la Data Governance (IR)

Le succès de la plateforme repose sur la consistance de ces fichiers pivots situés dans `.github/IR/` :

| Fichier | Rôle Stratégique |
| :--- | :--- |
| **`global.json`** | Source de vérité unique (Métadonnées, progression %, état des agents). |
| **`discovery_queue.json`** | File d'attente de traitement pour garantir la reprise après interruption. |
| **`executionTrace.json`** | Historique complet des actions, des patchs AST et des logs d'erreurs. |
| **`contracts/tests.json`** | Schéma des attentes comportementales immuables extraites de `/test`. |

---

> **Note de l'Architecte** : En utilisant Python pour le "Scout" déterministe et Babel pour l'écriture sémantique, nous transformons une migration risquée en un processus industriel contrôlé. L'humain devient le superviseur de l'**Intermediate Representation (IR)**.

Voici le **Guide de l'Utilisateur** pour ta plateforme de migration **V6 AST-Driven**. Ce guide est conçu pour t'accompagner depuis l'installation des dépendances jusqu'à la validation finale du clone de **react-grid-layout** (RGL).

---

## 🛠️ 1. Préparation de l'Environnement

Avant de lancer l'orchestrateur, tu dois t'assurer que le moteur de transformation (Babel) est prêt à opérer sur tes fichiers.

### Installation des Dépendances AST
Le moteur sémantique a besoin de bibliothèques spécifiques pour parser et transformer le code React/TypeScript :
1. Assure-toi d'avoir exécuté le fichier `setup-v6-ast-pipeline.js`.
2. Dans ton terminal, à la racine du projet, lance la commande suivante :
   `npm install @babel/parser @babel/traverse @babel/generator @babel/types`.

### Vérification des Fichiers de Base
Vérifie que les répertoires suivants ont été créés :
* **`.github/agents/`** : Contient tes 5 agents spécialisés.
* **`.github/skills/`** : Contient les protocoles d'analyse forensique.
* **`.github/IR/`** : Contient le fichier `global.json`, le cerveau de l'opération.

---

## 🚀 2. Lancement de la Migration

La plateforme fonctionne sur un modèle de **pilotage par l'orchestrateur**. Tu n'interagis quasiment qu'avec lui via le **Copilot CLI**.

### Étape 1 : Initialisation (Phase 0)
Ouvre ton Copilot CLI et tape la commande suivante :
> *"Utilise l'agent orchestrator. Initialise la Phase 0 : scanne /src et /test pour mapper les dépendances et créer la file d'attente dans l'IR."*

### Étape 2 : Analyse Forensique (Phases 1 & 2)
Une fois la cartographie terminée, demande à l'orchestrateur de lancer les experts :
> *"Lance l'analyse des tests et des composants. Extraits le contrat d'API et la logique DnD mathématique."*

**Ce qui va se passer :**
* Le `test-analyzer` va geler les comportements attendus (ex: compaction, collisions).
* Le `dnd-specialist` va isoler les fonctions de calcul (calcXY, etc.) pour les préparer à **dnd-kit 0.3.2**.

---

## 🧠 3. Supervision de l'IR (Intermediate Representation)

Contrairement à une discussion classique, tu peux surveiller la "pensée" de l'IA en ouvrant le fichier **`.github/IR/global.json`**.

| Section | Contenu à surveiller |
| :--- | :--- |
| **`component`** | Structure détectée (props, lifecycle React 18). |
| **`tests`** | Liste des garanties comportementales (le "Contrat d'Interface"). |
| **`dnd`** | Algorithmes extraits (Rising Tide, AABB). |
| **`tasks`** | Liste ordonnée des modifications AST à venir. |

---

## 🏗️ 4. Implémentation AST & Validation

C'est ici que la magie de la V6 opère. L'IA ne réécrit pas tes fichiers au hasard ; elle applique des **patchs sémantiques**.

### Exécution des Patchs
Lorsque l'architecture est validée dans l'IR, lance l'implémentation :
> *"Applique les patchs AST planifiés. Utilise le moteur Babel pour transformer src/core et src/react."*

### Boucle d'Auto-Correction
Si le moteur `ast-engine.js` rencontre une erreur de syntaxe dans le code généré par l'IA :
1. L'erreur sera logguée dans l'IR.
2. L'orchestrateur demandera automatiquement une correction au subagent `ast-implementer`.
3. Le patch sera retenté jusqu'à ce que l'AST soit valide.

---

## ✅ 5. Validation de Parité (Playwright)

La dernière étape consiste à prouver que le clone React 19 / dnd-kit se comporte exactement comme l'original.

1. L'agent `validation-agent` va lire les use-cases dans `test/examples/`.
2. Il va générer des scripts Playwright différentiels.
3. **Action Utilisateur** : Exécute les tests générés :
   `npx playwright test parity.spec.ts`

---

## 🛡️ Règles de Sécurité & Bonnes Pratiques

* **Ne modifie pas `global.json` manuellement** : L'IA s'appuie sur la structure exacte de ce fichier. Si tu veux corriger une analyse, demande-le à l'orchestrateur.
* **Le Dossier `/src/core/` est sacré** : C'est là que vivent tes mathématiques pures. Aucun code React (JSX/Hooks) ne doit y être injecté par l'IA.
* **Vérification des Ref** : En React 19, assure-toi que l'IA a bien supprimé les `forwardRef` pour utiliser les props `ref` standards.



**Prêt à transformer RGL ?** Commence par lancer la **Phase 0** avec ton orchestrateur !

} catch (e) { console.error('❌ Erreur Oracle : Assurez-vous que les tests passent avant capture.'); }

import os, json, hashlib
def get_hash(p): 
    with open(p, 'rb') as f: return hashlib.sha256(f.read()).hexdigest()

def analyze_v7():
    ir_p = '.github/IR/global.json'
    with open(ir_p, 'r') as f: data = json.load(f)
    ir_f, dep_g = {}, {}
    for r, _, files in os.walk('.'):
        for f in files:
            if f.endswith(('.ts', '.tsx', '.js', '.jsx')) and 'node_modules' not in r:
                p = os.path.join(r, f).replace('\\', '/')
                h = get_hash(p)
                with open(p, 'r', errors='ignore') as src: content = src.read()
                is_react = 'React' in content or 'useState' in content
                ir_f[p] = {
                    'state': 'discovered', 'currentHash': h,
                    'metadata': {'type': 'test' if '/test/' in p else ('react' if is_react else 'core_candidate')},
                    'last_agent': 'python-scout'
                }
    data['registry']['files'] = ir_f
    with open(ir_p, 'w') as f: json.dump(data, f, indent=2)
    print('✅ Identity Registry Initialized with SHA-256.')
analyze_v7()
